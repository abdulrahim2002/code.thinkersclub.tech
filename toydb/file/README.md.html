      2 
      3 [![CI](https://github.com/erikgrinaker/toydb/actions/workflows/ci.yml/badge.svg)](https://github.com/erikgrinaker/toydb/actions/workflows/ci.yml)
      4 
      5 Distributed SQL database in Rust, built from scratch as an educational project. Main features:
      6 
      7 * [Raft distributed consensus engine][raft] for linearizable state machine replication.
      8 
      9 * [ACID transaction engine][txn] with MVCC-based snapshot isolation.
     10 
     11 * [Pluggable storage engine][storage] with [BitCask][bitcask] and [in-memory][memory] backends.
     12 
     13 * [Iterator-based query engine][query] with [heuristic optimization][optimizer] and time-travel 
     14   support.
     15 
     16 * [SQL interface][sql] including joins, aggregates, and transactions.
     17 
     18 Originally written to learn more about database internals, toyDB is intended to illustrate the basic
     19 architecture and concepts of distributed SQL databases. It focuses on simplicity and 
     20 understandability, and should be functional and correct. Other aspects like performance,
     21 scalability, and availability are explicit non-goals -- these are major sources of complexity in 
     22 production-grade  databases, which obscure the basic underlying concepts. Shortcuts have been taken 
     23 where possible.

     44 ## Usage
     45 
     46 With a [Rust compiler](https://www.rust-lang.org/tools/install) installed, a local five-node 
     47 cluster can be built and started as:
     48 
     49 ```
     50 $ ./cluster/run.sh
     51 Starting 5 nodes on ports 9601-9605 with data under cluster/*/data/.
     52 To connect to node 5, run: cargo run --release --bin toysql
     53 
     54 toydb4 21:03:55 [INFO] Listening on [::1]:9604 (SQL) and [::1]:9704 (Raft)
     55 toydb1 21:03:55 [INFO] Listening on [::1]:9601 (SQL) and [::1]:9701 (Raft)
     56 toydb2 21:03:55 [INFO] Listening on [::1]:9602 (SQL) and [::1]:9702 (Raft)
     57 toydb3 21:03:55 [INFO] Listening on [::1]:9603 (SQL) and [::1]:9703 (Raft)
     58 toydb5 21:03:55 [INFO] Listening on [::1]:9605 (SQL) and [::1]:9705 (Raft)
     59 toydb2 21:03:56 [INFO] Starting new election for term 1
     60 [...]
     61 toydb2 21:03:56 [INFO] Won election for term 1, becoming leader
     62 ```
     63 
     64 A command-line client can be built and used with node 5 on `localhost:9605`:
     65 
     66 ```
     67 $ cargo run --release --bin toysql
     68 Connected to toyDB node n5. Enter !help for instructions.
     69 toydb> CREATE TABLE movies (id INTEGER PRIMARY KEY, title VARCHAR NOT NULL);
     70 toydb> INSERT INTO movies VALUES (1, 'Sicario'), (2, 'Stalker'), (3, 'Her');
     71 toydb> SELECT * FROM movies;
     72 1, 'Sicario'
     73 2, 'Stalker'
     74 3, 'Her'
     75 ```
     76 
     77 toyDB supports most common SQL features, including joins, aggregates, and transactions.
     78 
     79 Below is an `EXPLAIN` query plan of a more complex query, fetching movies from studios that have
     80 released movies with an IMDb rating of 8 or more:
     81 
     82 ```
     83 toydb> EXPLAIN SELECT m.title, g.name AS genre, s.name AS studio, m.rating
     84   FROM movies m JOIN genres g ON m.genre_id = g.id,
     85     studios s JOIN movies good ON good.studio_id = s.id AND good.rating >= 8
     86   WHERE m.studio_id = s.id
     87   GROUP BY m.title, g.name, s.name, m.rating, m.released
     88   ORDER BY m.rating DESC, m.released ASC, m.title ASC;
     89 
     90 Remap: m.title, genre, studio, m.rating (dropped: m.released)
     91 └─ Order: m.rating desc, m.released asc, m.title asc
     92    └─ Projection: m.title, g.name as genre, s.name as studio, m.rating, m.released
     93       └─ Aggregate: m.title, g.name, s.name, m.rating, m.released
     94          └─ HashJoin: inner on m.studio_id = s.id
     95             ├─ HashJoin: inner on m.genre_id = g.id
     96             │  ├─ Scan: movies as m
     97             │  └─ Scan: genres as g
     98             └─ HashJoin: inner on s.id = good.studio_id
     99                ├─ Scan: studios as s
    100                └─ Scan: movies as good (good.rating > 8 OR good.rating = 8)
    101 ```

    103 ## Architecture
    104 
    105 toyDB's architecture is fairly typical for a distributed SQL database: a transactional
    106 key/value store managed by a Raft cluster with a SQL query engine on top. See the
    107 [architecture guide](./docs/architecture.md) for more details.
    108

    111 ## Tests
    112 
    113 toyDB mainly uses [Goldenscripts](https://github.com/erikgrinaker/goldenscript) for tests. These 
    114 script various scenarios, capture events and output, and later assert that the behavior remains the 
    115 same. See e.g.:
    116 
    117 * [Raft cluster tests](https://github.com/erikgrinaker/toydb/tree/master/src/raft/testscripts/node)
    118 * [MVCC transaction tests](https://github.com/erikgrinaker/toydb/tree/master/src/storage/testscripts/mvcc)
    119 * [SQL execution tests](https://github.com/erikgrinaker/toydb/tree/master/src/sql/testscripts)
    120 * [End-to-end tests](https://github.com/erikgrinaker/toydb/tree/master/tests/scripts)
    121 
    122 Run tests with `cargo test`, or have a look at the latest 
    123 [CI run](https://github.com/erikgrinaker/toydb/actions/workflows/ci.yml).

    125 ## Benchmarks
    126 
    127 toyDB is not optimized for performance, but comes with a `workload` benchmark tool that can run 
    128 various workloads against a toyDB cluster. For example:
    129 
    130 ```sh
    131 # Start a 5-node toyDB cluster.
    132 $ ./cluster/run.sh
    133 [...]
    134 
    135 # Run a read-only benchmark via all 5 nodes.
    136 $ cargo run --release --bin workload read
    137 Preparing initial dataset... done (0.179s)
    138 Spawning 16 workers... done (0.006s)
    139 Running workload read (rows=1000 size=64 batch=1)...
    140 
    141 Time   Progress     Txns      Rate       p50       p90       p99      pMax
    142 1.0s      13.1%    13085   13020/s     1.3ms     1.5ms     1.9ms     8.4ms
    143 2.0s      27.2%    27183   13524/s     1.3ms     1.5ms     1.8ms     8.4ms
    144 3.0s      41.3%    41301   13702/s     1.2ms     1.5ms     1.8ms     8.4ms
    145 4.0s      55.3%    55340   13769/s     1.2ms     1.5ms     1.8ms     8.4ms
    146 5.0s      70.0%    70015   13936/s     1.2ms     1.5ms     1.8ms     8.4ms
    147 6.0s      84.7%    84663   14047/s     1.2ms     1.4ms     1.8ms     8.4ms
    148 7.0s      99.6%    99571   14166/s     1.2ms     1.4ms     1.7ms     8.4ms
    149 7.1s     100.0%   100000   14163/s     1.2ms     1.4ms     1.7ms     8.4ms
    150 
    151 Verifying dataset... done (0.002s)
    152 ```
    153 
    154 The available workloads are:
    155 
    156 * `read`: single-row primary key lookups.
    157 * `write`: single-row inserts to sequential primary keys.
    158 * `bank`: bank transfers between various customers and accounts. To make things interesting, this
    159   includes joins, secondary indexes, sorting, and conflicts.
    160 
    161 For more information about workloads and parameters, run `cargo run --bin workload -- --help`.
    162 
    163 Example workload results are listed below. Write performance is pretty atrocious, due to fsyncs 
    164 and a lack of write batching at the Raft level. Disabling fsyncs, or using the in-memory engine, 
    165 significantly improves write performance.
    166 
    167 | Workload | BitCask     | BitCask w/o fsync | Memory      |
    168 |----------|-------------|-------------------|-------------|
    169 | `read`   | 14163 txn/s | 13941 txn/s       | 13949 txn/s |
    170 | `write`  | 35 txn/s    | 4719 txn/s        | 7781 txn/s  |
    171 | `bank`   | 21 txn/s    | 1120 txn/s        | 1346 txn/s  |

    173 ## Debugging
    174 
    175 [VSCode](https://code.visualstudio.com) provides an intuitive environment for debugging toyDB.
    176 The debug configuration is included under `.vscode/launch.json`, to use it:
    177 
    178 1. Install the [CodeLLDB](https://marketplace.visualstudio.com/items?itemName=vadimcn.vscode-lldb)
    179    extension.
    180 
    181 2. Go to the "Run and Debug" tab and select e.g. "Debug unit tests in library 'toydb'".
    182 
    183 3. To debug the binary, select "Debug executable 'toydb'" under "Run and Debug".

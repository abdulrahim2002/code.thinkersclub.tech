      1 # toyDB Architecture
      2 
      3 At the highest level, toyDB consists of a cluster of nodes that execute SQL transactions against
      4 a replicated state machine. Clients can connect to any node in the cluster and submit SQL
      5 statements. It aims to provide
      6 [linearizability](https://jepsen.io/consistency/models/linearizable) (i.e. strong consistency)
      7 and [serializability](https://jepsen.io/consistency/models/serializable), but falls slightly
      8 short as it currently only implements
      9 [snapshot isolation](https://jepsen.io/consistency/models/snapshot-isolation).
     10 
     11 The [Raft algorithm](https://raft.github.io) is used for cluster consensus, which tolerates the
     12 failure of any node as long as a majority of nodes are still available. One node is elected
     13 leader, and replicates commands to the others which apply them to local copies of the state
     14 machine. If the leader is lost, a new leader is elected and the cluster continues operation.
     15 Client commands are automatically forwarded to the leader.
     16 
     17 This architecture guide will begin with a high-level overview of node components, before
     18 discussing each component from the bottom up. Along the way, we will make note of tradeoffs and
     19 design choices.

     37 ## Node Components
     38 
     39 A toyDB node consists of three main components:
     40 
     41 * **Storage engine:** stores data and manages transactions, on disk and in memory.
     42 
     43 * **Raft consensus engine:** handles cluster coordination and state machine replication.
     44 
     45 * **SQL engine:** parses, plans, and executes SQL statements for clients.
     46 
     47 These components are integrated in the toyDB server, which handles network communication with
     48 clients and other nodes. The following diagram illustrates its internal structure:
     49 
     50 ![toyDB architecture](./images/architecture.svg)
     51 
     52 At the bottom is a simple [key/value store](https://en.wikipedia.org/wiki/Key–value_database), 
     53 which stores all SQL data. This is wrapped inside an
     54 [MVCC](https://en.wikipedia.org/wiki/Multiversion_concurrency_control) key/value store that adds 
     55 [ACID transactions](https://en.wikipedia.org/wiki/ACID). On top of that is a SQL storage engine,
     56 providing basic [CRUD operations](https://en.wikipedia.org/wiki/Create,_read,_update_and_delete)
     57 on tables, rows, and indexes. This makes up the node's core storage engine.
     58 
     59 The SQL storage engine is wrapped in a Raft state machine interface, allowing it to be managed
     60 by the Raft consensus engine. The Raft node receives commands from clients and coordinates with
     61 other Raft nodes to reach consensus on an ordered command log. Once commands are committed to
     62 the log, they are applied to the local state machine.
     63 
     64 On top of the Raft engine is a Raft-based SQL storage engine, which implements the SQL storage
     65 interface and submits commands to the Raft cluster. This allows the rest of the SQL layer to use
     66 the Raft cluster as if it was local storage. The SQL engine manages client SQL sessions, which
     67 take SQL queries as text, parse them, generate query plans, and execute them against the SQL
     68 storage engine.
     69 
     70 Surrounding these components is the toyDB server, which in addition to network communication also
     71 handles configuration, logging, and other process-level concerns.
     72 
     73 ## Storage Engine
     74 
     75 ToyDB uses a pluggable key/value storage engine, with the SQL and Raft storage engines configurable
     76 via the `storage_sql` and `storage_raft` options respectively. The higher-level SQL storage engine 
     77 will be discussed separately in the [SQL section](#sql-engine).
     78 
     79 ### Key/Value Storage
     80 
     81 A key/value storage engine stores arbitrary key/value pairs as binary byte slices, and implements
     82 the
     83 [`storage::Engine`](https://github.com/erikgrinaker/toydb/blob/master/src/storage/engine.rs) 
     84 trait:
     85 
     86 ```rust
     87 /// A key/value storage engine, where both keys and values are arbitrary byte
     88 /// strings between 0 B and 2 GB, stored in lexicographical key order. Writes
     89 /// are only guaranteed durable after calling flush().
     90 ///
     91 /// Only supports single-threaded use since all methods (including reads) take a
     92 /// mutable reference -- serialized access can't be avoided anyway, since both
     93 /// Raft execution and file access is serial.
     94 pub trait Engine: std::fmt::Display + Send + Sync {
     95     /// The iterator returned by scan(). Traits can't return "impl Trait", and
     96     /// we don't want to use trait objects, so the type must be specified.
     97     type ScanIterator<'a>: DoubleEndedIterator<Item = Result<(Vec<u8>, Vec<u8>)>> + 'a
     98     where
     99         Self: 'a;
    100 
    101     /// Deletes a key, or does nothing if it does not exist.
    102     fn delete(&mut self, key: &[u8]) -> Result<()>;
    103 
    104     /// Flushes any buffered data to the underlying storage medium.
    105     fn flush(&mut self) -> Result<()>;
    106 
    107     /// Gets a value for a key, if it exists.
    108     fn get(&mut self, key: &[u8]) -> Result<Option<Vec<u8>>>;
    109 
    110     /// Iterates over an ordered range of key/value pairs.
    111     fn scan<R: std::ops::RangeBounds<Vec<u8>>>(&mut self, range: R) -> Self::ScanIterator<'_>;
    112 
    113     /// Sets a value for a key, replacing the existing value if any.
    114     fn set(&mut self, key: &[u8], value: Vec<u8>) -> Result<()>;
    115 }
    116 ```
    117 
    118 The `get`, `set` and `delete` methods simply read and write key/value pairs, and `flush` ensures
    119 any buffered data is written out to storage (e.g. via the `fsync` system call). `scan` iterates
    120 over a key/value range _in order_, a property that is crucial to higher-level functionality (e.g.
    121 SQL table scans) and has a couple of important implications:
    122 
    123 * Implementations should store data ordered by key, for performance.
    124 
    125 * Keys should use an order-preserving byte encoding, to allow range scans.
    126 
    127 The engine itself does not care what keys contain, but the storage module offers
    128 an order-preserving key encoding called [KeyCode](https://github.com/erikgrinaker/toydb/blob/master/src/encoding/keycode.rs)
    129 for use by higher layers. These storage layers often use composite keys made up
    130 of several possibly variable-length values (e.g. an index key consists of table,
    131 column, and value), and the natural ordering of each segment must be preserved,
    132 a property satisfied by this encoding:
    133 
    134 * `bool`: `0x00` for `false`, `0x01` for `true`.
    135 * `u64`: big-endian binary representation.
    136 * `i64`: big-endian binary representation, with sign bit flipped.
    137 * `f64`: big-endian binary representation, with sign bit flipped, and rest if negative.
    138 * `Vec<u8>`: `0x00` is escaped as `0x00ff`, terminated with `0x0000`.
    139 * `String`:  like `Vec<u8>`.
    140 
    141 Additionally, several container types are supported:
    142 
    143 * Tuple: concatenation of elements, with no surrounding structure.
    144 * Array: like tuple.
    145 * Vec: like tuple.
    146 * Enum: the variant's enum index as a single `u8` byte, then contents.
    147 * Value: like enum.
    148 
    149 The default key/value engine is
    150 [`storage::BitCask`](https://github.com/erikgrinaker/toydb/blob/master/src/storage/bitcask.rs),
    151 a very simple variant of Bitcask, an append-only log-structured storage engine.
    152 All writes are appended to a log file, with an index mapping live keys to file
    153 positions maintained in memory.  When the amount of garbage (replaced or deleted
    154 keys) in the file exceeds 20%, a new log file is written containing only live
    155 keys, replacing the old log file.
    156 
    157 #### Key/Value Tradeoffs
    158 
    159 **Keyset in memory:** BitCask requires the entire key set to fit in memory, and must also scan
    160 the log file on startup to construct the key index.
    161 
    162 **Compaction volume:** unlike an LSM tree, this single-file BitCask
    163 implementation requires rewriting the entire dataset during compactions, which
    164 can produce significant write amplification over time.
    165 
    166 **Key encoding:** does not make use of any compression, e.g. variable-length integers, preferring
    167 simplicity and correctness.
    168 
    169 ### MVCC Transactions
    170 
    171 [MVCC (Multi-Version Concurrency Control)](https://en.wikipedia.org/wiki/Multiversion_concurrency_control)
    172 is a relatively simple concurrency control mechanism that provides
    173 [ACID transactions](https://en.wikipedia.org/wiki/ACID) with
    174 [snapshot isolation](https://en.wikipedia.org/wiki/Snapshot_isolation) without taking out locks or 
    175 having writes block reads. It also versions all data, allowing querying of historical data.
    176 
    177 toyDB implements MVCC at the storage layer as
    178 [`storage::mvcc::MVCC`](https://github.com/erikgrinaker/toydb/blob/master/src/storage/mvcc.rs),
    179 using any `storage::Engine` implementation for underlying storage. `begin` returns a new
    180 transaction, which provides the usual key/value operations such as `get`, `set`, and `scan`.
    181 Additionally, it has a `commit` method which persists the changes and makes them visible to
    182 other transactions, and a `rollback` method which discards them.
    183 
    184 When a transaction begins, it fetches the next available version from `Key::NextVersion` and
    185 increments it, then records itself as an active transaction via `Key::TxnActive(version)`. It also
    186 takes a snapshot of the active set, containing the versions of all other active transactions as of
    187 the transaction start, and saves it as `Key::TxnActiveSnapshot(id)`.
    188 
    189 Key/value pairs are saved as `Key::Version(key, version)`, where `key` is the user-provided key
    190 and `version` is the transaction's version. The visibility of key/value pairs for a transaction is 
    191 given as follows:
    192 
    193 * For a given user key, do a reverse scan of `Key::Version(key, version)` starting at the current 
    194   transaction's version.
    195 
    196 * Skip any records whose version is in the transaction's snapshot of the active set.
    197 
    198 * Return the first matching record, if any. This record may be either a `Some(value)` or a `None`
    199   if the key was deleted.
    200 
    201 When writing a key/value pair, the transaction first checks for any conflicts by scanning for a
    202 `Key::Version(key, version)` which is not visible to it. If one is found, a serialization error
    203 is returned and the client must retry the transaction. Otherwise, the transaction writes the new
    204 record and keeps track of the change as `Key::TxnWrite(version, key)` in case it must roll back.
    205 
    206 When the transaction commits, it simply deletes its `Txn::Active(id)` record, thus making its
    207 changes visible to any subsequent transactions. If the transaction instead rolls back, it
    208 iterates over all `Key::TxnWrite(id, key)` entries and removes the written key/value records before
    209 removing its `Txn::Active(id)` entry.
    210 
    211 This simple scheme is sufficient to provide ACID transaction guarantees with snapshot isolation:
    212 commits are atomic, a transaction sees a consistent snapshot of the key/value store as of the
    213 start of the transaction, and any write conflicts result in serialization errors which must be
    214 retried.
    215 
    216 To satisfy time travel queries, a read-only transaction simply loads the `Key::TxnActiveSnapshot`
    217 entry of a past transaction and applies the same visibility rules as for normal transactions.
    218 
    219 #### MVCC Tradeoffs
    220 
    221 **Serializability:** snapshot isolation is not fully serializable, since it exhibits
    222 [write skew anomalies](http://justinjaffray.com/what-does-write-skew-look-like/). This would
    223 require [serializable snapshot isolation](https://courses.cs.washington.edu/courses/cse444/08au/544M/READING-LIST/fekete-sigmod2008.pdf),
    224 which was considered unnecessary for a first version - it may be implemented later.
    225 
    226 **Garbage collection:** old MVCC versions are never removed, leading to unbounded disk usage. 
    227 However, this also allows for complete data history, and simplifies the implementation.
    228 
    229 **Transaction ID overflow:** transaction IDs will overflow after 64 bits, but this is never going to
    230 happen with toyDB.

    232 ## Raft Consensus Engine
    233 
    234 The Raft consensus protocol is explained well in the
    235 [original Raft paper](https://raft.github.io/raft.pdf), and will not be repeated here - refer to
    236 it for details. toyDB's implementation follows the paper fairly closely.
    237 
    238 The Raft node [`raft::Node`](https://github.com/erikgrinaker/toydb/tree/master/src/raft/node.rs) is
    239 the core of the implementation, a finite state machine with enum variants for the node roles:
    240 leader, follower, and candidate. This enum wraps the `RawNode` struct, which contains common
    241 node functionality and is generic over the specific roles `Leader`, `Follower`, and `Candidate`
    242 that implement the Raft protocol.
    243 
    244 Nodes are initialized with an ID and a list of peer IDs, and communicate by passing
    245 [`raft::Message`](https://github.com/erikgrinaker/toydb/blob/master/src/raft/message.rs)
    246 messages. Inbound messages are received via `Node.step()` calls, and outbound messages are sent
    247 via an `mpsc` channel. Nodes also use a logical clock to keep track of e.g. election timeouts
    248 and heartbeats, and the clock is ticked at regular intervals via `Node.tick()` calls. These
    249 methods are synchronous and may cause state transitions, e.g. changing a candidate into a leader
    250 when it receives the winning vote.
    251 
    252 Nodes have a command log [`raft::Log`](https://github.com/erikgrinaker/toydb/blob/master/src/raft/log.rs),
    253 using a `storage::Engine` for storage, and a [`raft::State`](https://github.com/erikgrinaker/toydb/blob/master/src/raft/state.rs)
    254 state machine (the SQL engine). When the leader receives a write request, it appends the command
    255 to its local log and replicates it to followers. Once a quorum have replicated it, the command is
    256 committed and applied to the state machine, and the result returned the the client. When the leader
    257 receives a read request, it needs to ensure it is still the leader in order to satisfy 
    258 linearizability (a new leader could exist elsewhere resulting in a stale read). It increments a
    259 read sequence number and broadcasts it via a Raft heartbeat. Once a quorum have confirmed the
    260 leader at this sequence number, the read command is executed against the state machine and the
    261 result returned to the client.
    262 
    263 The actual network communication is handled by the server process, which will be described in a
    264 [separate section](#server).
    265 
    266 #### Raft Tradeoffs
    267 
    268 **Single-threaded state:** all state operations run in a single thread on the leader, preventing
    269 horizontal scalability. Improvements here would require running multiple sharded Raft clusters,
    270 which is out of scope for the project.
    271 
    272 **Synchronous application:** state machine application happens synchronously in the main Raft
    273 thread. This is significantly simpler than asynchronous application, but may cause delays in
    274 Raft processing.
    275 
    276 **Log replication:** only the simplest form of Raft log replication is implemented, without
    277 state snapshots or rapid log replay. Lagging nodes will be very slow to catch up.
    278 
    279 **Cluster resizing:** the Raft cluster consists of a static set of nodes given at startup, resizing
    280 it requires a complete cluster restart.

    282 ## SQL Engine
    283 
    284 The SQL engine builds on Raft and MVCC to provide a SQL interface to clients. Logically, the
    285 life of a SQL query is as follows:
    286 
    287 > Query → Lexer → Parser → Planner → Optimizer → Executor → Storage Engine
    288 
    289 We'll begin by looking at the basic SQL type and schema systems, as well as the SQL storage
    290 engine and its session interface. Then we'll switch sides and look at how a query is executed,
    291 starting at the front with the parser and following it until it's executed against the SQL
    292 storage engine, completing the chain.
    293 
    294 ### Types
    295 
    296 toyDB has a very simple type system, with the
    297 [`sql::DataType`](https://github.com/erikgrinaker/toydb/blob/master/src/sql/types/mod.rs) enum 
    298 specifying the available data types: `Boolean`, `Integer`, `Float`, and `String`.
    299 
    300 The [`sql::Value`](https://github.com/erikgrinaker/toydb/blob/master/src/sql/types/mod.rs) enum 
    301 represents a specific value using Rust's native type system, e.g. an integer value is 
    302 `Value::Integer(i64)`. This enum also specifies comparison, ordering, and formatting of values. The 
    303 special value `Value::Null` represents an unknown value of unknown type, following the rules of
    304 [three-valued logic](https://en.wikipedia.org/wiki/Three-valued_logic).
    305 
    306 Values can be grouped into a `Row`, which is an alias for `Vec<Value>`. The type `Rows` is an alias
    307 for a fallible row iterator, and `Column` is a result column containing a name.
    308 
    309 Expressions [`sql::Expression`](https://github.com/erikgrinaker/toydb/blob/master/src/sql/types/expression.rs)
    310 represent operations on values. For example, `(1 + 2) * 3` is represented as:
    311 
    312 ```rust
    313 Expression::Multiply(
    314     Expression::Add(
    315         Expression::Constant(Value::Integer(1)),
    316         Expression::Constant(Value::Integer(2)),
    317     ),
    318     Expression::Constant(Value::Integer(3)),
    319 )
    320 ```
    321 
    322 Calling `evaluate()` on the expression will recursively evaluate it, returning `Value::Integer(9)`.
    323 
    324 ### Schemas
    325 
    326 The schema defines the tables [`sql::Table`](https://github.com/erikgrinaker/toydb/blob/master/src/sql/schema.rs)
    327 and columns [`sql::Column`](https://github.com/erikgrinaker/toydb/blob/master/src/sql/schema.rs)
    328 in a toyDB database. Tables have a name and a list of columns, while a column has several
    329 attributes such as name, data type, and various constraints. They also have methods to
    330 validate rows and values, e.g. to make sure a value is of the correct type for a column
    331 or to enforce referential integrity.
    332 
    333 The schema is stored and managed with [`sql::Catalog`](https://github.com/erikgrinaker/toydb/blob/master/src/sql/schema.rs),
    334 a trait implemented by the SQL storage engine:
    335 
    336 ```rust
    337 pub trait Catalog {
    338     /// Creates a new table.
    339     fn create_table(&mut self, table: &Table) -> Result<()>;
    340 
    341     /// Deletes a table, or errors if it does not exist.
    342     fn delete_table(&mut self, table: &str) -> Result<()>;
    343 
    344     /// Reads a table, if it exists.
    345     fn read_table(&self, table: &str) -> Result<Option<Table>>;
    346 
    347     /// Iterates over all tables.
    348     fn scan_tables(&self) -> Result<Tables>;
    349 }
    350 ```
    351 
    352 #### Schema Tradeoffs
    353 
    354 **Single database:** only a single, unnamed database is supported per toyDB cluster. This is
    355 sufficient for toyDB's use-cases, and simplifies the implementation.
    356 
    357 **Schema changes:** schema changes other than creating or dropping tables is not supported. This
    358 avoids complicated data migration logic, and allows using table/column names as storage identifiers 
    359 (since they can never change) without any additional indirection.
    360 
    361 ### Storage
    362 
    363 The SQL storage engine trait is [`sql::Engine`](https://github.com/erikgrinaker/toydb/blob/master/src/sql/engine/mod.rs):
    364 
    365 ```rust
    366 pub trait Engine: Clone {
    367     type Transaction: Transaction;
    368 
    369     /// Begins a transaction in the given mode.
    370     fn begin(&self, mode: Mode) -> Result<Self::Transaction>;
    371 
    372     /// Resumes an active transaction with the given ID.
    373     fn resume(&self, id: u64) -> Result<Self::Transaction>;
    374 
    375     /// Begins a SQL session for executing SQL statements.
    376     fn session(&self) -> Result<Session<Self>> {
    377         Ok(Session { engine: self.clone(), txn: None })
    378     }
    379 }
    380 ```
    381 
    382 The main use of the trait is to dispense `sql::Session` instances, individual client sessions which
    383 execute SQL queries submitted as plain text and track transaction state. The actual storage
    384 engine functionality is exposed via the `sql::Transaction` trait, representing an ACID transaction
    385 providing basic CRUD (create, read, update, delete) operations for tables, rows, and indexes:
    386 
    387 ```rust
    388 pub trait Transaction: Catalog {
    389     /// Commits the transaction.
    390     fn commit(self) -> Result<()>;
    391     /// Rolls back the transaction.
    392     fn rollback(self) -> Result<()>;
    393 
    394     /// Creates a new table row.
    395     fn create(&mut self, table: &str, row: Row) -> Result<()>;
    396     /// Deletes a table row.
    397     fn delete(&mut self, table: &str, id: &Value) -> Result<()>;
    398     /// Reads a table row, if it exists.
    399     fn read(&self, table: &str, id: &Value) -> Result<Option<Row>>;
    400     /// Scans a table's rows, optionally filtering by the given predicate expression.
    401     fn scan(&self, table: &str, filter: Option<Expression>) -> Result<Scan>;
    402     /// Updates a table row.
    403     fn update(&mut self, table: &str, id: &Value, row: Row) -> Result<()>;
    404 
    405     /// Reads an index entry, if it exists.
    406     fn read_index(&self, table: &str, column: &str, value: &Value) -> Result<HashSet<Value>>;
    407     /// Scans a column's index entries.
    408     fn scan_index(&self, table: &str, column: &str) -> Result<IndexScan>;
    409 }
    410 ```
    411 
    412 The main SQL storage engine implementation is
    413 [`sql::engine::KV`](https://github.com/erikgrinaker/toydb/blob/master/src/sql/engine/kv.rs), which 
    414 is built on top of an MVCC key/value store and its transaction functionality.
    415 
    416 The Raft SQL storage engine
    417 [`sql::engine::Raft`](https://github.com/erikgrinaker/toydb/blob/master/src/sql/engine/raft.rs)
    418 uses a Raft API client `raft::Client` to submit state machine commands specified by the enums 
    419 `Mutation` and `Query` to the local Raft node. It also provides a Raft state machine 
    420 `sql::engine::raft::State` which wraps a regular `sql::engine::KV` SQL storage engine and applies 
    421 state machine commands to it. Since the Raft SQL engine implements the `sql::Engine` trait, it can 
    422 be used interchangably with the local storage engine.
    423 
    424 #### Storage Tradeoffs
    425 
    426 **Raft result streaming:** result streaming is not implemented for Raft commands, so the Raft
    427 SQL engine must buffer the entire result set in memory and serialize it before returning it to
    428 the client - particularly expensive for table scans. Implementing streaming in Raft was considered 
    429 out of scope for the project.
    430 
    431 ### Parsing
    432 
    433 The SQL session [`sql::Session`](https://github.com/erikgrinaker/toydb/blob/master/src/sql/engine/mod.rs)
    434 takes plain-text SQL queries via `execute()` and returns the result. The first step in this process 
    435 is to parse the query into an [abstract syntax tree](https://en.wikipedia.org/wiki/Abstract_syntax_tree)
    436 (AST) which represents the query semantics. This happens as follows:
    437 
    438 > SQL → Lexer → Tokens → Parser → AST
    439 
    440 The lexer
    441 [`sql::Lexer`](https://github.com/erikgrinaker/toydb/blob/master/src/sql/parser/lexer.rs) takes
    442 a SQL string, splits it into pieces, and classifies them as tokens `sql::Token`. It does not
    443 care about the meaning of the tokens, but removes whitespace and tries to figure out if
    444 something is a number, string, keyword, and so on. It also does some basic pre-processing, such as
    445 interpreting string quotes, checking number formatting, and rejecting unknown keywords.
    446 
    447 For example, the following input string results in the listed tokens, even though the query is 
    448 invalid:
    449 
    450 > `3.14 +UPDATE 'abc'` → `Token::Number("3.14")` `Token::Plus` `Token::Keyword(Keyword::Update)` `Token::String("abc")`
    451 
    452 The parser [`sql::Parser`](https://github.com/erikgrinaker/toydb/blob/master/src/sql/parser/mod.rs)
    453 iterates over the tokens generated by the lexer, interprets them, and builds an AST representing
    454 the semantic query. For example, `SELECT name, 2020 - birthyear AS age FROM people` 
    455 results in the following AST:
    456 
    457 ```rust
    458 ast::Statement::Select{
    459     select: vec![
    460         (ast::Expression::Field(None, "name"), None),
    461         (ast::Expression::Operation(
    462             ast::Operation::Subtract(
    463                 ast::Expression::Literal(ast::Literal::Integer(2020)),
    464                 ast::Expression::Field(None, "birthyear"),
    465             )
    466         ), Some("age")),
    467     ],
    468     from: vec![
    469         ast::FromItem::Table{name: "people", alias: None},
    470     ],
    471     where: None,
    472     group_by: vec![],
    473     having: None,
    474     order: vec![],
    475     offset: None,
    476     limit: None,
    477 }
    478 ```
    479 
    480 The parser will interpret the SQL _syntax_, determining the type of query and its parameters,
    481 returning an error for any invalid syntax. However, it has no idea if the table `people`
    482 actually exists, or if the column `birthyear` is an integer - that is the job of the planner.
    483 
    484 Notably, the parser also parses expressions, such as `1 + 2 * 3`. This is non-trivial due to
    485 precedence rules, i.e. `2 * 3` should be evaluated first, but not if there are parentheses
    486 around `(1 + 2)`. The toyDB parser uses the
    487 [precedence climbing algorithm](https://eli.thegreenplace.net/2012/08/02/parsing-expressions-by-precedence-climbing)
    488 for this. Also note that AST expressions are different from SQL engine expressions, and do not map 
    489 one-to-one. This is clearest in the case of function calls, where the parser does not know (or 
    490 care) if a given function exists, it just parses a function call as an arbitrary function name and 
    491 arguments. The planner will translate this into actual expressions that can be evaluated.
    492 
    493 ### Planning
    494 
    495 The SQL planner [`sql::Planner`](https://github.com/erikgrinaker/toydb/blob/master/src/sql/plan/planner.rs)
    496 takes the AST generated by the parser and builds a SQL execution plan
    497 [`sql::Plan`](https://github.com/erikgrinaker/toydb/blob/master/src/sql/plan/mod.rs), which is an 
    498 abstract representation of the steps necessary to execute the query. For example, the following 
    499 shows a simple query and corresponding execution plan, formatted as `EXPLAIN` output:
    500 
    501 ```
    502 SELECT id, title, rating * 100 FROM movies WHERE released > 2010 ORDER BY rating DESC;
    503 
    504 Order: rating desc
    505 └─ Projection: id, title, rating * 100
    506    └─ Filter: released > 2010
    507       └─ Scan: movies
    508 ```
    509 
    510 The plan nodes `sql::Node` in a query execution plan represent a
    511 [relational algebra](https://en.wikipedia.org/wiki/Relational_algebra) operator, where the
    512 output from one node flows as input into the next. In the example above, the query first does a
    513 full table scan of the `movies` table, then applies the filter `released > 2010` to the rows,
    514 before projecting (formatting) the result and sorting it by `rating`.
    515 
    516 Most of the planning is fairly straightforward, translating AST nodes to plan nodes and expressions.
    517 The trickiest part is resolving table and column names to result column indexes across multiple
    518 layers of aliases, joins, and projections - this is handled with `sql::plan::Scope`, which keeps
    519 track of what names are visible to the node being built and maps them to column indexes. Another
    520 challenge is aggregate functions, which are implemented as a pre-projection of function
    521 arguments and grouping/hidden columns, then an aggregation node, and finally a post-projection
    522 evaluating the final aggregate expressions - like in the following example:
    523 
    524 ```
    525 SELECT   g.name AS genre, MAX(rating * 100) - MIN(rating * 100)
    526 FROM     movies m JOIN genres g ON m.genre_id = g.id
    527 WHERE    m.released > 2000
    528 GROUP BY g.id, g.name
    529 HAVING   MIN(rating) > 7 
    530 ORDER BY g.id ASC;
    531 
    532 Projection: #0, #1
    533 └─ Order: g.id asc
    534    └─ Filter: #2 > 7
    535       └─ Projection: genre, #0 - #1, #2, g.id
    536          └─ Aggregation: maximum, minimum, minimum
    537             └─ Projection: rating * 100, rating * 100, rating, g.id, g.name
    538                └─ Filter: m.released > 2000
    539                   └─ NestedLoopJoin: inner on m.genre_id = g.id
    540                      ├─ Scan: movies as m
    541                      └─ Scan: genres as g
    542 ```
    543 
    544 The planner generates a very naïve execution plan, primarily concerned with producing one that
    545 is _correct_ but not necessarily _fast_. This means that it will always do full table scans,
    546 always use [nested loop joins](https://en.wikipedia.org/wiki/Nested_loop_join), and so on. The plan 
    547 is then optimized by a series of optimizers implementing
    548 [`sql::Optimizer`](https://github.com/erikgrinaker/toydb/blob/master/src/sql/plan/optimizer.rs):
    549 
    550 * `ConstantFolder`: pre-evaluates constant expressions to avoid having to re-evaluate them for each 
    551   row.
    552 
    553 * `FilterPushdown`: pushes filters deeper into the query to reduce the number of rows evaluated by
    554   each node, e.g. by pushing single-table predicates all the way to the table scan node such that
    555   filtered nodes won't have to go across the Raft layer.
    556 
    557 * `IndexLookup`: transforms table scans into primary key or index lookups where possible.
    558 
    559 * `NoopCleaner`: attempts to remove noop operations, e.g. filter nodes that evaluate to a constant 
    560   `TRUE` value.
    561 
    562 * `JoinType`: transforms nested loop joins into hash joins for equijoins (equality join predicate).
    563 
    564 Optimizers make heavy use of [boolean algebra](https://en.wikipedia.org/wiki/Boolean_algebra) to
    565 transform expressions into forms that are more convenient to work with. For example, partial
    566 filter pushdown (e.g. across join nodes) can only push down conjunctive clauses (i.e. AND parts),
    567 so expressions are converted into
    568 [conjunctive normal form](https://en.wikipedia.org/wiki/Conjunctive_normal_form) first such that
    569 each part can be considered separately.
    570 
    571 Below is an example of a complex optimized plan where table scans have been replaced with
    572 key and index lookups, filters have been pushed down into scan nodes, and nested loop joins have
    573 been replaced by hash joins. It fetches science fiction movies released since 2000 by studios
    574 that have released any movie with a rating of 8 or more:
    575 
    576 ```
    577 SELECT   m.id, m.title, g.name AS genre, m.released, s.name AS studio
    578 FROM     movies m JOIN genres g ON m.genre_id = g.id,
    579          studios s JOIN movies good ON good.studio_id = s.id AND good.rating >= 8
    580 WHERE    m.studio_id = s.id AND m.released >= 2000 AND g.id = 1
    581 ORDER BY m.title ASC;
    582 
    583 Order: m.title asc
    584 └─ Projection: m.id, m.title, g.name, m.released, s.name
    585    └─ HashJoin: inner on m.studio_id = s.id
    586       ├─ HashJoin: inner on m.genre_id = g.id
    587       │  ├─ Filter: m.released > 2000 OR m.released = 2000
    588       │  │  └─ IndexLookup: movies as m column genre_id (1)
    589       │  └─ KeyLookup: genres as g (1)
    590       └─ HashJoin: inner on s.id = good.studio_id
    591          ├─ Scan: studios as s
    592          └─ Scan: movies as good (good.rating > 8 OR good.rating = 8)
    593 ```
    594 
    595 #### Planning Tradeoffs
    596 
    597 **Type checking:** expression type conflicts are only detected at evaluation time, not during 
    598 planning.

    600 ### Execution
    601 
    602 Every SQL plan node has a corresponding executor, implementing the
    603 [`sql::Executor`](https://github.com/erikgrinaker/toydb/blob/master/src/sql/execution/mod.rs) trait:
    604 
    605 ```rust
    606 pub trait Executor<T: Transaction> {
    607     /// Executes the executor, consuming it and returning a result set
    608     fn execute(self: Box<Self>, txn: &mut T) -> Result<ResultSet>;
    609 }
    610 ```
    611 
    612 Executors are given a `sql::Transaction` to access the SQL storage engine, and return a 
    613 `sql::ResultSet` with the query result. Most often, the result is of type `sql::ResultSet::Query` 
    614 containing a list of columns and a row iterator. Most executors contain other executors that
    615 they use as inputs, for example the `Filter` executor will often have a `Scan` executor as a source:
    616 
    617 ```rust
    618 pub struct Filter<T: Transaction> {
    619     source: Box<dyn Executor<T>>,
    620     predicate: Expression,
    621 }
    622 ```
    623 
    624 Calling `execute` on a `sql::Plan` will build and execute the root node's executor, which in
    625 turn will recursively call `execute` on its source executors (if any) and process their results.
    626 Executors typically augment the source's returned row iterator using Rust's
    627 [`Iterator`](https://doc.rust-lang.org/std/iter/trait.Iterator.html) functionality, e.g. by
    628 calling `filter()` on it and returning a new iterator. The entire execution engine thus works in
    629 a streaming fashion and leverages Rust's [zero-cost iterator
    630 abstractions](https://doc.rust-lang.org/book/ch13-04-performance.html).
    631 
    632 Finally, the root `ResultSet` is returned to the client.

    634 ## Server
    635 
    636 The toyDB [`Server`](https://github.com/erikgrinaker/toydb/blob/master/src/server.rs) manages 
    637 network traffic for the Raft and SQL engines, using the [Tokio](https://tokio.rs) async executor. 
    638 It opens TCP listeners on port `9605` for SQL clients and  `9705` for Raft peers, both using 
    639 length-prefixed [Bincode](https://github.com/servo/bincode)-encoded message passing via
    640 [Serde](https://serde.rs)-encoded Tokio streams as a protocol.
    641 
    642 The Raft server is split out to [`raft::Server`](https://github.com/erikgrinaker/toydb/blob/master/src/raft/server.rs),
    643 which runs a main [event loop](https://en.wikipedia.org/wiki/Event_loop) routing Raft messages 
    644 between the local Raft node, TCP peers, and local state machine clients (i.e. the Raft SQL engine 
    645 wrapper), as well as ticking the Raft logical clock at regular intervals. It spawns separate Tokio 
    646 tasks that maintain outbound TCP connections to all Raft peers, while internal communication 
    647 happens via `mpsc`channels.
    648 
    649 The SQL server spawns a new Tokio task for each SQL client that connects, running a separate
    650 SQL session from the SQL storage engine on top of Raft. It communicates with the client by passing
    651 `server::Request` and `server::Response` messages that are translated to `sql::Session` calls.
    652 
    653 The main [`toydb`](https://github.com/erikgrinaker/toydb/blob/master/src/bin/toydb.rs) binary
    654 simply initializes a toyDB server based on command-line arguments and configuration files, and then 
    655 runs it via the Tokio runtime.
    656 
    657 #### Server Tradeoffs
    658 
    659 **Security:** all network traffic is unauthenticated an in plaintext, as security was considered
    660 out of scope for the project.

    662 ## Client
    663 
    664 The toyDB [`Client`](https://github.com/erikgrinaker/toydb/blob/master/src/client.rs) provides a 
    665 simple API for interacting with a server, mainly by executing SQL statements via `execute()` 
    666 returning `sql::ResultSet`.
    667 
    668 The [`toysql`](https://github.com/erikgrinaker/toydb/blob/master/src/bin/toysql.rs) command-line
    669 client is a simple REPL client that connects to a server using the toyDB `Client` and continually 
    670 prompts the user for a SQL query to execute, displaying the returned result.
